### `carton`

- `idpartida`, nullable, como referencia histórica;
- `indicevictoria`, nullable, como dato histórico;
- dueño, código, matriz, precio, fecha y estado del maestro.

| Campo | Mapeo físico |

- `UNIQUE(idcarton, idpartida)`;
- FK compuesta `(idcarton, idbingo)`;
- FK compuesta `(idpartida, idbingo)`;
- cartón y partida pertenecientes al mismo Bingo.

## 4. Diferencia entre el modelo antiguo y el objetivo

```text
ACTUAL
Carton --idpartida--> Partidabingo --idbingo--> Bingo
  └── indicevictoria global

OBJETIVO
Carton maestro --idbingo-----------------------> Bingo
       |
       +--< CartonPartidaBingo >-- Partidabingo
                 ├── estado por ronda
                 ├── indicevictoria por ronda
                 └── resultado por ronda
```

En el modelo objetivo:

- código, matriz, dueño, precio y compra existen una sola vez;
- el precio se cobra y reporta una sola vez por maestro;
- la participación y el resultado pertenecen a una ronda;
- un cartón puede quedar `Ganador` en varias partidas del mismo Bingo;
- `Carton.idpartida` y `Carton.indicevictoria` nunca son fuente de verdad para
  cartones nuevos.

## 5. Primer cambio requerido en `models.py`

El primer cambio de código debe ser el mapeo exacto del esquema expandido en
`apps/bingos/models.py`. No se deben crear migraciones Django.

### Cambios de `Carton`

1. Agregar `idbingo = models.ForeignKey(...)` con `db_column='idbingo'` y un
   `related_name` explícito, por ejemplo `cartones`.
2. Mantener `idpartida` e `indicevictoria` con sus nombres y columnas actuales
   durante la compatibilidad.
3. Marcar ambos como históricos mediante comentarios y excluirlos de todos los
   formularios de alta nuevos.
4. Los cartones nuevos deben guardar `idpartida=NULL` e
   `indicevictoria=NULL`; solo `idbingo` representa su pertenencia.
5. Mantener `managed = False` y `db_table = 'carton'`.

### Nuevo `CartonPartidaBingo`

Mapeo propuesto:

```python
class CartonPartidaBingo(models.Model):
    idcartonpartidabingo = models.AutoField(primary_key=True)
    idcarton = models.ForeignKey(
        "bingos.Carton",
        models.DO_NOTHING,
        db_column="idcarton",
        related_name="participaciones",
    )
    idpartida = models.ForeignKey(
        "bingos.Partidabingo",
        models.DO_NOTHING,
        db_column="idpartida",
        related_name="participaciones_carton",
    )
    idbingo = models.ForeignKey(
        "bingos.Bingo",
        models.DO_NOTHING,
        db_column="idbingo",
        related_name="participaciones_carton",
    )
    estado_participacion = models.CharField(max_length=20)
    indicevictoria = models.IntegerField(blank=True, null=True)
    es_asignacion_original = models.BooleanField(default=False)
    origen_asignacion = models.CharField(max_length=24)
    motivoestado = models.CharField(max_length=255, blank=True, null=True)
    fechacreacion = models.DateTimeField()
    fechavalidacion = models.DateTimeField(blank=True, null=True)

    class Meta:
        managed = False
        db_table = "carton_partida_bingo"
        unique_together = (("idcarton", "idpartida"),)
```

`AutoField` es obligatorio para que Django trate la PK `IDENTITY` como
autogenerada y recupere el ID insertado. No debe usarse
`assign_next_integer_pk` para esta tabla.

Django no representa aquí las dos FK compuestas como relaciones compuestas.
`idbingo` debe mapearse igualmente y los servicios deben validar igualdad; las
constraints físicas siguen siendo la garantía final.

## 6. Conservación de campos históricos

Durante la fase compatible:

- lecturas históricas pueden mostrar `Carton.idpartida` como trazabilidad, pero
  el flujo funcional debe consultar `CartonPartidaBingo`;
- `Carton.indicevictoria` no se actualiza nunca más;
- las 12 filas migradas tienen `es_asignacion_original=True`;
- los cartones nuevos dejan ambos campos históricos en `NULL`;
- ninguna consulta nueva puede usar `Carton.objects.filter(idpartida=...)` para
  decidir participación;
- la retirada física de ambos campos corresponde a una fase posterior, después
  de demostrar que no quedan lecturas ni escrituras.

Una capa de consultas centralizada debe evitar que cada vista implemente joins
distintos. Como mínimo debe ofrecer:

- participaciones de una partida con maestro y jugador precargados;
- participaciones de un cartón ordenadas por ronda;
- cartones maestros de un Bingo sin duplicación;
- participación exacta para la pareja cartón/partida.

## 7. Generación de cartones nuevos

La operación debe ser atómica y pertenecer a la capa de servicios:

1. recibir un Bingo, jugador, precio y fecha, no una única partida como dueño;
2. bloquear la fila de Bingo y obtener con bloqueo estable sus partidas;
3. validar que la venta todavía está permitida;
4. generar una sola matriz y un solo código, preferiblemente con prefijo de
   Bingo en lugar de partida;
5. crear un solo `Carton` con `idbingo`, precio único e históricos en `NULL`;
6. crear exactamente una `CartonPartidaBingo` por cada partida existente, con
   `origen_asignacion='Aplicacion'`;
7. no copiar precio ni matriz a las participaciones;
8. revertir todo si falla una sola participación.

Si existen `N` partidas, deben quedar un maestro y `N` participaciones. La
recaudación aumenta una sola vez.

Debe definirse además el caso inverso: si se permite agregar una partida a un
Bingo después de vender cartones, `partida_nueva` debe crear, en la misma
transacción, una participación para cada maestro existente. Ambos servicios
deben bloquear el Bingo en el mismo orden para evitar carreras.

La alternativa más segura es cerrar la venta cuando alguna ronda ya empezó. De
lo contrario, crear un cartón después de rondas finalizadas produciría filas
para rondas pasadas y volvería a introducir una inferencia histórica.

## 9. Adaptación funcional por flujo

### Consola del operador

- consultar participaciones de la partida y precargar maestro/jugador;
- mostrar estado e índice de la participación, no el índice histórico;
- validar mediante la pareja cartón/partida;
- editar por separado datos maestros y datos de ronda;
- mantener generación por Bingo, aunque se llegue desde una ronda.

### Extracción de bolas

- conservar bolas y última bola en `Partidabingo`;
- calcular progreso usando la matriz maestra y bolas de la ronda seleccionada;
- no escribir progreso global en `Carton`;
- publicar solo el grupo WebSocket de esa partida.

### Validación de ganador

- comprobar existencia de `CartonPartidaBingo` para la pareja;
- bloquear partida, participaciones candidatas y maestros necesarios;
- evaluar matrices maestras contra bolas de esa partida;
- marcar la participación exacta como `Ganador` y fecha de validación;
- definir y guardar `indicevictoria` positivo por ronda;
- permitir que el mismo `idcarton` gane otra partida del mismo Bingo.

### Desempate

- persistir en candidatos `idcartonpartidabingo`, además de cartón y jugador;
- no perder varios cartones del mismo jugador;
- al confirmar, actualizar la participación exacta ganadora;
- conservar `Partidabingo.idjugadorganador` como resumen compatible del jugador.

### Mis cartones

- listar una fila por maestro, agrupada por Bingo;
- no repetir precio o matriz por ronda;
- detalle con selector/lista de participaciones;
- progreso y estado calculados para la ronda seleccionada;
- mantener control de propiedad por `Carton.idjugador`.

### Vista pública

- resolver el código global a un maestro;
- mostrar Bingo y una lista controlada de rondas;
- exigir que la ronda seleccionada tenga una participación del cartón;
- elegir de forma determinista la ronda inicial: activa, luego pendiente y por
  último la cerrada más reciente;
- no inferir participación desde `Carton.idpartida`.

### PDF y Excel

- reporte de partida: filas de participación con datos del maestro;
- índice/estado desde la participación;
- resumen de Bingo: maestros y recaudación únicos, participaciones por ronda;
- ganador desde la participación, sin inferirlo por propietario;
- conservar exclusión de datos privados.

## 11. Casos de prueba obligatorios antes de tocar `bingo`

1. El mapeo unmanaged lee los 12 maestros y 12 participaciones históricas.
2. `AutoField` inserta una participación y recupera su IDENTITY sin calcular
   `MAX(id)+1`.
3. Un Bingo con `N` partidas crea un maestro y exactamente `N` participaciones.
4. Un fallo en una participación revierte maestro, precio y todas las filas.
5. La matriz y el código existen una sola vez; el precio se suma una sola vez.
6. Una partida agregada después crea una fila para cada maestro existente, si
   esa operación queda permitida.
7. Dos altas concurrentes no duplican código, PK ni pareja cartón/partida.
8. Las FK compuestas rechazan mezclar Bingos.
9. `UNIQUE(idcarton,idpartida)` rechaza duplicados.
10. Los cartones nuevos dejan históricos `idpartida` e `indicevictoria` en
    `NULL`.
11. Los 12 cartones antiguos siguen visibles solo en su participación original.
12. Un cartón puede ganar dos rondas distintas del mismo Bingo.
13. Validar una ronda no cambia resultado ni progreso de otra.
14. Varios ganadores generan desempate con la participación exacta.
15. Mis cartones lista maestros una sola vez y protege propiedad.
16. La consulta pública no permite seleccionar una ronda de otro Bingo/cartón.
17. PDF y Excel de Bingo mantienen recaudación `41.00`/`51.00` en el histórico
    y no multiplican precios por participación.
18. La consola, extracción y WebSocket afectan solo la ronda seleccionada.
19. Los payloads públicos no filtran datos privados.
20. `python manage.py check` sigue limpio y no aparece ninguna migración Django
    para modelos `managed=False`.

Las pruebas con base deben ejecutarse sobre una restauración aislada o una base
de ensayo recreable, nunca sobre `bingo`.

## 12. Riesgos restantes

- Semántica exacta de `indicevictoria`: debe fijarse si representa número de
  bolas extraídas, orden de validación u otra métrica positiva.
- Ventana de venta: debe impedir crear filas para rondas ya jugadas.
- Partidas creadas después de vender cartones: requiere backfill transaccional
  o prohibición explícita.
- `Partidabingo.idjugadorganador` guarda jugador, no participación; el detalle
  exacto debe permanecer en la intermedia.
- El admin genérico y formularios directos pueden evadir invariantes.

- La base real todavía no posee las columnas/tablas que exigirán los modelos

## 13. Estrategia de despliegue final

La migración física y el despliegue del código híbrido requieren una ventana
sin escrituras de cartones. En cuanto `carton.idbingo` pasa a `NOT NULL`, el
código anterior deja de poder crear cartones: su `INSERT` no envía `idbingo` y
PostgreSQL lo rechaza. Que las columnas nuevas no rompan ciertas lecturas del
código antiguo no convierte ese código en compatible para escritura.

1. Completar código y pruebas sobre `bingo_ensayo_hibridos` migrada.
2. Preparar artefacto de aplicación compatible con el esquema expandido.
3. Tomar respaldo y restaurarlo con éxito.
4. Abrir ventana y detener, como mínimo, todas las escrituras de cartones y la
   creación de partidas que pueda generar participaciones.
5. Confirmar `SELECT current_database()` y ejecutar la migración histórica en
   `bingo` solo con autorización independiente.
6. Validar 12/12/0 y controles de integridad.
7. Desplegar inmediatamente el código híbrido, sin reactivar el flujo anterior
   de creación de cartones durante el intervalo.
8. Ejecutar checks, smoke tests presenciales/virtuales, generación de un cartón
   controlado, consola, ganador, Mis cartones y reportes.
9. Reabrir escrituras únicamente si todos los controles pasan.
10. Observar y reconciliar maestros, participaciones y recaudación.
11. Retirar `Carton.idpartida` y `Carton.indicevictoria` solo en una etapa
    posterior, cuando búsquedas estáticas, métricas y pruebas demuestren que ya
    no se usan.

Si falla algo antes de escrituras híbridas, se debe mantener la ventana cerrada,
restaurar el código anterior y aplicar el rollback ensayado antes de permitirle
crear cartones otra vez. Después de escrituras nuevas se requiere una reversión
específica que preserve maestros y participaciones; no se debe usar el rollback
simple.

compra y precio proceden de `Carton`; ronda, estado de participación, índice
de victoria y fecha de validación proceden de `CartonPartidaBingo`.

Cada participación híbrida se valida contra su maestro, ronda y Bingo. Una
relación inconsistente no se mezcla con los datos válidos y produce una
respuesta administrativa controlada. Una participación de otra ronda se
excluye del reporte actual, sin inferir la ronda desde `Carton.idpartida` ni
usar `Carton.indicevictoria` como resultado híbrido.

### 21.1 PDF y Excel por partida

El PDF de partida conserva el resumen de Bingo, ronda, bolas, ganador y
desempate, y agrega una tabla discreta de cartones. Cada fila identifica
`Histórico por partida` o `Cartón de Bingo` y muestra código, jugador, estado
del maestro, estado de participación, índice, fecha de validación y si ganó
esa ronda. No presenta una victoria de ronda como victoria de todo el Bingo.

El Excel de cartones por partida incluye una fila por cartón histórico y una
por participación híbrida válida. Agrega columnas para tipo de registro,
estado de participación, fecha de validación y ganador de la ronda. Un mismo
maestro puede aparecer correctamente en los archivos de rondas diferentes,
una vez por su participación concreta, sin tomar resultados de otra ronda.

### 21.2 Excel resumen por Bingo

La hoja `Resumen de partidas` agrega por ronda los cartones históricos y las
participaciones híbridas, conservando la información administrativa existente.
La hoja nueva `Cartones del Bingo` muestra una sola fila por maestro. Código,
dueño, precio, matriz y fecha de compra no se repiten por cada participación.

Cada maestro híbrido informa número de participaciones, rondas ganadas, rondas
pendientes o activas y resumen de estados. Los totales de cartones y
recaudación del Bingo se calculan sobre maestros únicos. Los cartones
históricos mantienen su relación y resultado heredados sin inventar estados
de participación.

## 22. Etapa 9.6D.5 — Tiempo real por partida y participación

El contrato de tiempo real conserva un único grupo público por partida, con el
nombre `partida_<idpartidabingo>`, y la ruta
`/ws/juego/partidas/<idpartidabingo>/`. No se crean grupos por Bingo, cartón,
jugador ni participación. El consumidor comprueba que la partida exista,
agrega el cliente exclusivamente a ese grupo e ignora comandos enviados por el
cliente.

Las publicaciones de bola, inicio, pausa, reanudación, finalización, ganador y
desempate continúan usando la `Partidabingo` concreta operada por la vista o
devuelta por el servicio. La validación híbrida publica la ronda validada y no
obtiene el destino desde `Carton.idpartida` o `Carton.indicevictoria`. El flujo
histórico mantiene el mismo grupo y los mismos eventos por partida.

El cliente toma la ronda de `data-partida-id`, construye el WebSocket con ese
ID y descarta defensivamente cualquier payload cuyo `partida.id` no coincida.
Así, una actualización de la ronda B no modifica una pantalla conectada a la
ronda A, aunque ambas compartan el mismo maestro.

### 22.1 Selección híbrida y recarga segura

Las actualizaciones ordinarias de bolas y estado continúan aplicándose sobre
el DOM sin recargar. Los eventos `ganador_detectado`, `desempate_detectado` y
`desempate_finalizado` incluyen ahora `requiere_recarga=true`, porque sus datos
de participación deben volver a consultarse en el servidor.

El JavaScript recarga mediante `window.location.reload()` únicamente después
de validar el ID de la partida. La recarga conserva la ruta, el query string y
el fragmento actuales; por tanto, un detalle privado o público abierto con
`?partida=<idpartida>` mantiene la misma participación seleccionada. No se
construyen redirecciones ni URLs alternativas.

### 22.2 Privacidad del payload público

El payload conserva su tipo `partida_actualizada`, evento, ID de partida,
estado, mensajes públicos, bolas, última bola, conteos, finalización y bandera
de desempate. Cuando existe un ganador final solo publica el valor genérico
`Confirmado`; ya no transmite alias o nombre del jugador.

Los eventos no incluyen precio, correo, alias, nombre, ID de jugador, matriz,
código de otros cartones, candidatos de desempate ni campos internos de
`CartonPartidaBingo`. La firma de publicación se mantiene compatible con los
emisores existentes, pero un nombre recibido no se incorpora al payload.
